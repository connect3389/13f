# 美股 13F 自动化分析框架 — 需求与方法总纲

> 版本：v0.5（需求整理阶段）  
> 更新：**数据源统一为 SEC EDGAR**，删除第三方聚合站重复爬取；SQLite / 分析 / HTML 等后续阶段不变。

---

## 1. 背景与目标

### 1.1 业务目标

构建一套可重复运行的 **13F 持仓分析框架**，支持：

- 从 **SEC EDGAR**（`data.sec.gov/submissions` + `Archives` 目录下 **13F-HR / 13F-HR/A** 及 `index.json` 解析出的信息表 XML）自动化拉取并入库；访问规范见 [SEC 开发者说明](https://www.sec.gov/developer)。
- 使用 **Jupyter Notebook**（结合 `numpy`/pandas 等）完成清洗、校验与特征构造；
- **多机构横向比对**：持仓重叠、同向/反向变动、集中度与拥挤度等；
- **穿透分析能力**（见 §5）：在合规与技术可行范围内，将表层持仓（ETF、ADR、包装证券）映射到更底层的经济敞口或行业/因子结构。
- **HTML 报告输出**（见 §3.1）：在分析流水线末端生成交互式或半交互式 HTML（含柱状图、条形图、折线、热力或桑基等——按阶段选用），便于归档、邮件分享或静态托管；必要时与 **Notebook 内绘图** 共用同一套指标定义。
- **SQLite 本地历史库**（见 §3.2）：对已披露的 13F 历史快照做持久化；**重复抓取时若本地已存在完整记录则跳过网络拉取与解析**，从而快速「中断」无效工作、降费源与站点压力。修正案等例外见 §3.2。

### 1.2 非目标（当前阶段明确排除或降级）

- 不构成投资建议；不做实时交易信号引擎的硬承诺（13F 天然滞后，见 §4）。
- 不默认解决所有衍生品经济敞口的精确复原（披露盲区见 §4）。

## 2. 用户场景与功能需求

### 2.1 角色与典型流程

| 角色 | 行为 |
|------|------|
| 分析师 | 选定机构清单、季度区间、关注标的池；跑 Notebook 生成对比表、图表及 **导出的 HTML 报告** |
| 运营/你本人 | 维护 `filers_watchlist`、`.env`（SEC UA）与定时/手动抓取任务；观察失败与重试 |

### 2.2 功能性需求（FR）

- **FR-01 EDGAR 批量抓取**：读取 **`filers_watchlist`**（§2.4），按 **CIK** 调用 SEC 公开接口拉取 **13F-HR(/A)**，尊重 **声明式 User-Agent** 与请求间隔；**抓取前查询 SQLite（§3.2）**，对已 `complete` 且版本满足的报告 **跳过**（`--force` 可覆盖）。不向第三方聚合站做重复爬取。
- **FR-02 数据抽取与规范化**：将 EDGAR XML（含 `index.json` 引导的辅助文件）解析为统一 schema（见 §7）。
- **FR-03 清洗校验**：CUSIP/名称对齐、重复行合并、空值与单位（报告期末市值单位等）一致性检查。
- **FR-04 多机构比对**：同一报告期末窗口内，多 filer 的持仓集合与权重对比；季度间变动对比。
- **FR-05 穿透（分阶段）**：ETF 成分分解（需第三方持仓数据源或本地成分库）、行业映射（GICS 等）、ADR 与底层市场提示。
- **FR-06 可追溯**：保留原始 XML 路径、SHA256、解析器版本、抓取时间戳；可选保留调试截图。
- **FR-07 HTML 报告**：每个分析批次可产出 **至少一份主 HTML**（可多页或单页分区），包含：报告期/数据来源说明、§4.4 风险摘要、核心表（如 Top 持仓、QoQ 变动分类统计）、**已嵌入的图表**（bar 图等）；文件路径与命名规则可配置（如 `reports/13f_{quarter}_{run_id}.html`）。
- **FR-08 SQLite 持久化与增量策略**：使用本地 **SQLite**（默认路径可配置，如 `data/13f_history.sqlite`）保存：(1) 每次抓取/解析的 **作业元数据**；(2) 规范化后的 **持仓行**（与 §7.1 对齐）；(3) 原始 XML 路径与 **内容哈希**（可选存 blob，默认仅存路径）。支持 **`--force`** 或配置项覆盖跳过逻辑以强制重拉。**可中断**：批处理在「跳过已完备」的机构或季度后可直接进入下一项或正常结束，无需清空历史。

### 2.3 非功能性需求（NFR）

- **合规**：遵守 **[SEC 公平访问政策](https://www.sec.gov/developer)**（User-Agent、频率上限等）；**唯一权威持仓来源为 EDGAR**，不通过其它网站重复抓取同一份 13F。
- **可维护**：解析层与业务分析层分离（爬虫/导出解析 ≠ Notebook 内的统计逻辑）。
- **可复现**：固定季度截止日、数据版本、成分表版本（用于穿透）。
- **报告体积**：交互式 HTML（尤其内嵌整库 JS 时）单文件可能较大；里程碑早期可接受 ≤ 数 MB，日后可通过 CDN 引用或拆分为 **数据 JSON + 精简页面** 优化。
- **本地存储**：SQLite 文件纳入 `.gitignore`（若含路径隐私）；定期备份；可选用 **WAL** 模式降低 Notebook 与爬虫并发读写的锁冲突。
- **安全**：`.env` 不入库；勿把 SEC UA 或密钥写进 Notebook 输出与报告。

### 2.4 跟踪机构清单（`filers_watchlist`，EDGAR / CIK）

- **定义**：与代码分离的配置（如 `config/filers_watchlist.yaml`），每一行至少 **`cik`**（程序可补零至 10 位）；可选 `display_name`、`notes`、`style_tags`，供展示与分类。
- **轮询**：批任务按清单顺序对各 **CIK** 拉取 `submissions` 中最近若干份 **13F-HR(/A)**；增删机构 **只改清单**。
- **主键**：全链路以 **CIK** 标识机构；人工若只有机构英文名，可先用 [SEC Company Search](https://www.sec.gov/edgar/searchedgar/companysearch) 查 CIK。

## 3. 推荐技术架构

```
┌─────────────────────────────────────────────────────────┐
│  抓取：CLI / 定时任务 → filers_watchlist（CIK）            │
│        → data.sec.gov + Archives（含 index.json 辅文件）   │
└───────────────────────────┬─────────────────────────────┘
                            ▼
┌─────────────────────────────────────────────────────────┐
│  解析层：Python；信息表 XML → 统一 schema；质量与审计字段   │
└───────────────────────────┬─────────────────────────────┘
                            ▼
┌─────────────────────────────────────────────────────────┐
│  持久层：SQLite — 作业状态、持仓行、raw 哈希、parser 版本   │
└───────────────────────────┬─────────────────────────────┘
                            ▼
┌─────────────────────────────────────────────────────────┐
│  分析层：Jupyter — 特征、比对、穿透、可视化               │
└───────────────────────────┬─────────────────────────────┘
                            ▼
┌─────────────────────────────────────────────────────────┐
│  报告层：HTML（Plotly 等）                               │
└─────────────────────────────────────────────────────────┘
```

**说明**：单一数据平面 **EDGAR**，避免多源重复与不覆盖。

### 3.1 HTML 报告与可视化栈（推荐组合）

**目标**：从同一份 pandas 结果表快速得到 **bar / 折线 / 热力** 等图，并写入 **独立 HTML**，无需先搭完整商业 BI。

**关于「shacdn / shadcn」**  
主流名称是 **[shadcn/ui](https://ui.shadcn.com/)**（基于 React + Tailwind CSS 的 **UI 组件**库：按钮、表格、卡片、布局等）。它 **不是** 图表库；条形图、柱状图通常再配 **[Recharts](https://recharts.org/)**、**[Chart.js](https://www.chartjs.org/)**（react-chartjs-2）或 **Tremor** 等。

因此需求上拆两条路线（可并存：阶段 1 用 A，阶段 2 加 B）：

| 路线 | 适用场景 | 图表与 HTML | 与 Notebook 关系 |
|------|-----------|-------------|------------------|
| **A. Python 优先（建议先做）** | 分析、回测、定时任务全系在 Python | **[Plotly](https://plotly.com/python/)**（`fig.write_html()` 单文件交互图）、或 **[Altair](https://altair-viz.github.io/)** / Vega-Lite 导出 HTML；表格可用 `pandas.Styler.to_html()` 或 Jinja2 拼模板 | Notebook 或 `reports/generate_report.py` 直接产出 HTML |
| **B. shadcn 小站（可选）** | 需要精选仪表盘、品牌化 UI、多端复用 | **Next.js + shadcn/ui** 做壳与表；**Recharts** 等做 bar/line；Python pipeline 导出 **`report_data.json`**（或 Parquet → 构建时静态生成）供前端读取 | Python 负责数据；前端负责展示与路由 |

**本框架的默认约定（v0.3）**

- **MVP**：采用 **路线 A**，以 **Plotly** 为主力（柱状/条形/堆叠、时间序列、热力图成熟，且一份 `write_html` 即可分享）。  
- **增强**：若你希望报告像产品站点（筛选器、多 Tab、暗色主题），再引入 **路线 B**；shadcn 负责版式与组件，**不负责** 从 DataFrame 直接出图。

**报告内容清单（最低限度）**

1. 元信息：生成时间、`filers_watchlist` 版本或 CIK 列表、**数据来源：SEC EDGAR**。  
2. 合规短声明：链回 §4.4（滞后、多头披露等）。  
3. 图表建议：**权重条形图（Top N）**、**QoQ 股数变动分类计数**、**机构间重叠度（热力）**、**行业占比堆叠条（穿透 L1 后）**。

### 3.2 SQLite 本地存储与重复抓取「跳过 / 早退」

**动机**：已提交的 **Form 13F-HR** 对应季末持仓在监管意义上是 **历史快照**，通常不会像行情那样被「改写」；实务上重复全量拉取同一季报收益有限。**本地 SQLite** 用于沉淀历史抓取与解析结果，并**记录**校验哈希、解析器版本、原始文件路径等，使下一轮任务可 **O(1) 判定是否已完备** 并 **中断** 对该 filer/季度的后续网络与解析步骤。

**例外（须在设计中显式处理）**

- **13F-HR/A 修正案**：可能更正往期信息；库中需能区分 `form_type` / `is_amendment` / `accession_number`，必要时 **提高 (cik, report_date) 维度优先级** 给较新 accession，或对修正案触发 `invalidate` + 重拉。  
- **本站或 EDGAR 侧行级纠错**：若 `raw_sha256` 与上次不同且非修正案，应记日志并可选自动重解析。

**跳过判定（建议默认策略）**

在发起 **HTTP** 拉取前，对已解析入库的 `(filer_cik, report_date, source_channel, ingest_kind)` 查询：

- 若存在 `ingest_record.status = 'complete'` 且 `holdings_row_count > 0`（或与官方行数门槛一致）、`parser_version` 不低于当前要求，则 **跳过抓取**；  
- 若存在 `failed` / `partial`，由配置决定：**重试** 或 **跳过需人工**。  
- CLI 标志 **`--force`** / 配置 **`refresh_policy: always|missing_only`** 覆盖上述行为。

**建议在库中记录的信息（除 §7.1 持仓行外）**

| 类别 | 示例字段 | 用途 |
|------|----------|------|
| 作业追溯 | `run_id`, `started_at`, `finished_at`, `hostname`, `watchlist_hash` | 复现哪次批跑 |
| 来源 | `source_channel`（如 `edgar_xml`） | 与实现一致 |
| 原始工件 | `raw_path`, `raw_sha256`, `downloaded_at` | 去重、取证 |
| 解析 | `parser_version`, `parse_ok`, `row_count`, `warnings_json` | 跳过与回归 |
| 实体 | `filer_cik`、`report_date`（季末） | 主检索维度 |

**SQLite 物理表草案（与 §7.3 对应，实现时可微调）**

- `filer_registry`：CIK、展示名、与 watchlist 同步。  
- `ingest_record`：一次「某 filer × 某 report_date × 某 source」的抓取/解析单元；**唯一约束**建议含 `source_channel` 与 `report_date`。  
- `holding_line`：规范化明细行；或通过 `ingest_id` 外键关联。  
- 可选 `ingest_run`：批任务头表，多行 `ingest_record` 指向同一 `run_id`。

**与 Notebook / 报告**

- Notebook **默认从 SQLite 读取** 已规范化的 `holding_line`，减少重复清洗；仅在 `refresh` 后重算。  
- HTML 报告（FR-07）可引用 `--run_id` 或最新 `complete` 快照。

## 4. 核心分析逻辑：当前市场通行的 13F「理论」与边界

本节是需求文档中的 **核心逻辑底座**：实务分析应默认在「监管披露语义」下解读，而非等同于基金经理的真实实时组合。

### 4.1 13F 在监管上是什么（摘要）

- **Form 13F** 是美国《证券交易法》下的机构持仓披露工具，立法目的包括提升大型机构持仓的公共可得性（SEC 官方 FAQ 对制度背景有说明）。[SEC Form 13F FAQs](https://www.sec.gov/rules-regulations/staff-guidance/division-investment-management-frequently-asked-questions/frequently-asked-questions-about-form-13f)
- **触发条件（概念层）**：对 Section 13(f) 证券行使投资自主权且规模达到规则门槛的 institutional investment manager 需报送（FAQ 对「谁必须报」有定义）。[同上](https://www.sec.gov/rules-regulations/staff-guidance/division-investment-management-frequently-asked-questions/frequently-asked-questions-about-form-13f)
- **报告内容（概念层）**：信息表一般包括发行人名称、证券类别描述、持股数量、报告期末公允市值等（FAQ 对报表要素有列举）。[同上](https://www.sec.gov/rules-regulations/staff-guidance/division-investment-management-frequently-asked-questions/frequently-asked-questions-about-form-13f)
- **2022 年修正案要点（实施层面）**：自 2023-01-03 起使用修订版表格；例如可增加 FIGI、调整市值取整规则等（SEC Release 与 FAQ 有索引）。[SEC FAQ 更新说明](https://www.sec.gov/rules-regulations/staff-guidance/division-investment-management-frequently-asked-questions/frequently-asked-questions-about-form-13f)

### 4.2 实务解读框架：以「季度间变动」为中心

公开市场材料普遍强调：**单期快照信息量有限，季度间（QoQ）变动的解释力更强**。典型将持仓变动分为四类并分档解读信号强弱（机构投研科普向，非监管文本）：

1. **新开仓（New）**：常被视为较高信号；需结合 **占组合权重** 再看「信念强度」。  
2. **增持（Increase）**：持续性看好；区分「显著加仓」与小幅再平衡。  
3. **减持（Trim）**：单一季度最难解读；**多季度轨迹** 更有意义。  
4. **清仓（Sold out）**：强负面/退出信号，但仍需排除换仓到同类标的或包装工具。

参考综述性读物（非 SEC）：  
- [How to Read SEC 13F Filings（面向投资者的说明文）](https://www.heygotrade.com/en/blog/how-to-read-sec-13f-filings/)  
- [Reading 13F Changes: New Buys, Sells & Position Changes | HedgeTrace](https://hedgetrace.com/learn/reading-institutional-changes)  
- [How to Track Hedge Fund Portfolios Using 13F Data | 13F Insight](https://13finsight.com/learn/how-to-track-hedge-fund-portfolios-13f-data-guide)

### 4.3 分析原则（写入本框架的强制约束）

- **优先比较股数，再参考市值**：季度之间股价波动会扭曲「市值变动」叙述；**股数变动**更接近真实交易方向（实务共识，见 HedgeTrace / 13F Insight 类文章）。
- **仓位必须情境化**：同一标的在不同 AUM、策略（被动/量化/集中多头/长短期）下的含义不同；分析前应有 **经理分类标签**（数据字典见 §7.2）。
- **拥挤度与会众**：多单机在同标的上的同向增减持，可形成「拥挤交易」提示；需避免机械跟单（13F Insight 等对 crowd 有专门讨论）。[Crowded Institutional Trades | 13F Insight](https://13finsight.com/learn/how-to-read-crowded-institutional-trades-without-chasing-them)
- **集中度指标**：单标的权重、Top N 占比、HHI（赫芬达尔）等，用于比较机构间「赌注结构」差异。

### 4.4 披露盲区与误读来源（必须在产品上显性提示）

下列内容适用于向终端用户展示分析结论时的 **风险提示模块**：

- **滞后**：季度末后至公开通常存在法定期限内的披露延迟，读到报告时持仓可能已显著变化（科普向总结见 HedgeTrace 等）。[13F Limitations | HedgeTrace](https://hedgetrace.com/learn/13f-limitations)
- **偏多头可见性**：13F 无法完整反映空头、全部衍生品结构、现金比例等；长空仓并举时，多头持仓可能被空头对冲（「盲区别」类文章常用表述）。[What Form 13F Doesn't Capture | 13F Insight](https://13finsight.com/learn/what-form-13f-doesnt-capture-shorts-options-blind-spots)
- **期权/互换等经济敞口与报表行项目语义**：需单独建模，不能默认等同于股票 delta。  
- **实体类型噪声**：托管行聚合、做市商库存等可能扭曲「信念」解读（实务讨论见 13F Insight 等）。  

### 4.5 建议的「核心指标清单」（分析层 MVP）

| 指标 | 定义要点 | 用途 |
|------|----------|------|
| QoQ ΔShares | 同 CUSIP 股数差分 | 主叙事线 |
| QoQ ΔWeight | 期末市值占比差分 | 组合内重要性变化 |
| 行为分类 | New / Add / Trim / Exit | 故事线与筛选 |
| 组合重叠 | Jaccard、重叠权重和 | 机构「集群」 |
| 同向性 | 多机构在同一标的上的方向一致比例 | 拥挤度近似 |
| 集中度 | Top 10 weight、HHI | 风险赌注对比 |

## 5. 「穿透」需求定义（分层落地）

为避免范围失控，将穿透拆成 **L0–L3**：

| 层级 | 内容 | 数据依赖 |
|------|------|----------|
| L0 | 证券层穿透：CUSIP/ISIN/FIGI 归一 + 名称别称 | 映射表 |
| L1 | 行业/板块：映射至 GICS（或自选行业体系） | 行业分类表 |
| L2 | ETF 持仓穿透：将 ETF 持仓按最新持仓披露分解为底层资产敞口 | ETF 持仓数据源（如基金公司披露/API/本地库） |
| L3 | 因子/主题：构建简化因子暴露（价值/动量/规模等） | 日频或季频因子库（可选，阶段二） |

**需求决策点**：L2 以 **报告期末可得的最近一次 ETF 持仓** 近似，还是允许 **当季末同步重构**，需在实现前固定规则并在 Notebook 中写明版本。

## 6. 清单文件草案（`filers_watchlist`）

示例（字段名可调整）：

```yaml
# config/filers_watchlist.yaml — 仅 SEC CIK 驱动抓取
defaults:
  refresh_policy: missing_only

filers:
  - cik: "0002045724"
    display_name: Situational Awareness Lp
    style_tags: [hf, concentrated]
  # - cik: "0001067983"
  #   display_name: Berkshire Hathaway Inc
```

## 7. 数据模型（建议最小 schema）

### 7.1 `holding_snapshot`（持仓快照，一行一 filer–标的–季度）

| 字段 | 说明 |
|------|------|
| `filer_id` | 机构稳定标识（CIK/CRD/自建 UUID） |
| `report_date` | 报告期末（季末） |
| `line_no` | 原表行号（若有） |
| `issuer` | 发行人名称 |
| `title_of_class` | 证券类别描述 |
| `cusip` | 主键优先 |
| `figi` | 若有（2022 后更常见） |
| `shares` | 股数 |
| `value_usd` | 报告期末市值（USD；注意历史规则中取整方式） |
| `weight` | `value / sum(value)` |
| `source` | `edgar_xml`（或与代码枚举一致） |
| `ingested_at` | 入库时间 |

### 7.2 `filer_meta`（机构元数据）

- `style_tags`：passive / active / bank / hf / allocator…  
- `aum_estimate_band`（可选）  
- `notes`：结构复杂性（长空仓、高期权用量等）

### 7.3 SQLite 表与 §7.1 的映射（摘要）

| SQLite 表（建议名） | 对应逻辑 | 备注 |
|---------------------|----------|------|
| `filer_registry` | CIK、展示名 | 与 `filers_watchlist` 同步 |
| `ingest_record` | 一次抓取+解析的元数据 | `status`, `raw_sha256`, `parser_version`；**跳过逻辑的主查询表** |
| `holding_line` | `holding_snapshot` 行 | 外键 `ingest_id`；联合唯一可考虑 `(ingest_id, cusip, line_no)` |

详细 DDL 留在实现仓库；需求层只要求 **唯一性约束** 与 **索引**（`(filer_cik, report_date)`、`ingest_record.status`）支撑快速跳过。
